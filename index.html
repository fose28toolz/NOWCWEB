Definition and characteristics of ELB

Elastic Load Balancing

●     Ensuring that no single Amazon EC2 instance has to carry the full workload on its own.

●     Exists as a managed service engineered to address the undifferentiated heavy lifting of load balancing

●     It runs at the regional level and is a high-performance, cost-efficient, highly available, and automatically scalable system.

●     Uses Auto Scaling Groups to register with the regional ELB

 

 

Service Description and Example

ELB is automatically scalable. As your traffic grows, ELB is designed to handle the additional throughput with no change to the hourly cost.

When your EC2 fleet auto-scales out (more instances come online for increased demand), the auto-scaling service lets the Elastic Load Balancing service know that a new instance is ready to handle the traffic. Once registered with the ELB, the ELB can then send traffic over to the new instance, and off it goes.

When the EC2 fleet auto-scales in (instances prep to shutdown as demand decreases), ELB first stops all new traffic and waits for the existing requests to be completed and to drain out. This means the current transactions will be completed, but there will be no advertisements for new available connections. Once they do that, the auto-scaling engine can terminate the instances without disrupting existing customers.

 

The fleet is the resource that I create and assign to an auto-scaling group.

The ELB uses assigned metrics to monitor the fleet instances. If there are many requests to the load balancer, the load balancer can distribute the new request to the scaled-up instances.

 

Elastic Load Balancing creates the instances but scales them as they register availability to service the requests/workloads.

 

When scaling the fleet, a new request never assigned to an instance scheduled for termination.  Finish what you are working on before you die/terminate.

Four types of Load balancers are provided by Elastic Load Balancing

Application Load Balancer

Routes and load balances at the application layer (HTTP/HTTPS) and support path-based routing

An Application Load Balancer can route requests to ports on one or more registered targets, such as EC2 instances, in your virtual private cloud (VPC).

Network Load Balancer

Routes and load balances at the transport layer (TCP/UDP Layer-4), based on address information extracted from the Layer-4 header.

Network Load Balancers can handle traffic bursts, retain the client's source IP, and use a fixed IP for the load balancer's lifetime.

Gateway Load Balancer

Distributes traffic to a fleet of appliance instances.

It provides scale, availability, and simplicity for third-party virtual appliances, such as firewalls, intrusion detection and prevention systems.

Gateway Load Balancers work with virtual appliances that support the GENEVE protocol. Additional technical integration is required, so consult the user guide before choosing a Load Balancer.

Classic Load Balancer

Routes and load balances either at the transport layer (TCP/SSL), or at the application layer (HTTP/HTTPS).

With Classic Load Balancers, instances are registered directly with the load balancer.

 

 

There is a key difference in how the load balancer types are configured. With Application Load Balancers, Network Load Balancers, and Gateway Load Balancers, instances are registered as targets with a target group, and you route traffic to the target group. With Classic Load Balancers, instances are registered directly with the load balancer.

Well, we solve the back-end traffic chaos with an ELB as well. Because ELB is regional, each front-end instance uses a single URL. Then, the ELB directs traffic to the back end with the fewest outstanding requests. If the back end scales, once the new instance is ready, it just tells the ELB that it can take traffic and get to work. The front end doesn't know or care how many back-end instances are running. This is authentic decoupled architecture.

 

Monolithic applications and microservices

Monolithic applications

Monolithic applications are indicated by tightly coupled components. Applications are made of multiple components. These components communicate with each other to transmit data, fulfill requests, and keep the application running.

 

In this approach to application architecture, if a single component fails, other components fail, and possibly the entire application fails.

 

 

 

Microservices approach

In a microservices approach, application components are loosely coupled. if a single component fails, the other components continue to work because they communicate. The loose coupling prevents the entire application from failing.

 

When designing applications on AWS, you can take a microservices approach with services and components that fulfill different functions.

 

 

 

Two services facilitate application integration: Amazon Simple Notification Service (Amazon SNS) and Amazon Simple Queue Service (Amazon SQS).

Messaging and Queuing

Amazon Simple Queue Service or SQS

Allows you to send, store, and receive messages between software components at any volume without losing messages or requiring other services to be available.

The data contained within a message is called a payload, and it's protected until delivery.

SQS queues are where messages are placed until they are processed.

AWS manages the underlying infrastructure so you can host those queues.

The queues scale automatically, are reliable and are easy to configure and use.

 

Amazon Simple Notification Service or SNS

Amazon SNS is similar in that it sends messages to services and notifications to end users. However, it does this in a different way, called a publish/subscribe or pub/sub model.

  1. This means you can create an SNS topic, which is just a channel for delivering messages.
  2. You then configure subscribers to that topic and finally publish messages for those subscribers.

 

In practice one, you can send one message to a topic, which will then spread to all the subscribers in a single go.

 

These subscribers can also be endpoints like SQS queues, AWS Lambda functions, and HTTPS or HTTP webhooks.

Additionally, SNS can fan out notifications to end users using mobile push, SMS, and email. Notification, such as when a customer's order is ready or a status is updated.

Definitions

 

Tenancy In AWS

Tenancy determines who owns a resource and how EC2 instances run on hardware:

Shared tenancy

The default option is where multiple customers share the same hardware. This is the cheapest option for running EC2 instances.

Dedicated tenancy

Ensures that EC2 instances run on hardware specific to your account. There are two types of dedicated tenancy:

●     Dedicated instances: Your instance runs on single-tenant hardware, meaning no other AWS customer shares the same physical server.

●     Dedicated hosts: You purchase an entire physical host from AWS, which you can use to spin up EC2 instances. Dedicated hosts give you more visibility and control over how instances are placed on a physical server. Dedicated Hosts allow customers to bring their licenses (BYOL) with dedicated hardware requirements.
